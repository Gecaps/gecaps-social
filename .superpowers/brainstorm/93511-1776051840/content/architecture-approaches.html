Arquitetura do Social Content OS

3 abordagens para reestruturar o gecaps-social seguindo o PRD. Clique na que preferir.

A

Modular por Dominio (Recomendada)

Estrutura: Cada modulo do PRD vira uma pasta isolada com seus proprios componentes, actions, types e hooks.

app/
  (conta)/[accountId]/
    metricas/          -- dashboard da conta
    calendario/        -- calendario editorial
    pipeline/          -- kanban de pecas
    referencias/       -- biblioteca de referencias
    ideias/            -- ideias geradas/manuais
    pecas/[id]/        -- editor de peca
    publicados/        -- historico publicado
    playbook/          -- brand playbook
    configuracoes/
  api/
    references/        -- CRUD referencias
    ideas/             -- geracao + CRUD ideias
    pieces/            -- geracao + CRUD pecas
    accounts/          -- CRUD contas
    ai/generate/       -- endpoint unificado IA
    render/            -- Cloudflare render HD
    metrics/           -- metricas manuais
modules/
  references/          -- logica de dominio
  ideas/
  pieces/
  playbook/
  metrics/
  ai/                  -- prompts, few-shot, pipeline IA
lib/                   -- clients (supabase, cloudflare, etc)

Pros

  • Cada modulo e independente e testavel
  • Facil de entender onde cada coisa mora
  • Escala bem pra SaaS (adicionar modulos sem bagunca)
  • Rotas por conta via [accountId] ja isolam contexto

Cons

  • Mais pastas/arquivos que o projeto atual
  • Precisa refatorar a estrutura existente
B

Feature Slices (Vertical)

Estrutura: Cada feature e uma fatia vertical completa (UI + API + logica + types) numa unica pasta.

features/
  references/
    components/        -- UI
    actions/           -- server actions
    api/               -- route handlers
    lib/               -- logica de dominio
    types.ts
  ideas/
    components/
    actions/
    api/
    lib/
    types.ts
  pieces/
    ...
app/
  (conta)/[accountId]/
    metricas/page.tsx      -- importa de features/
    calendario/page.tsx
    pipeline/page.tsx
    referencias/page.tsx

Pros

  • Tudo de uma feature num lugar so
  • Maximo de coesao por feature
  • Facil deletar/substituir um modulo inteiro

Cons

  • Foge do padrao Next.js App Router (APIs fora de app/api)
  • Mais complexo de configurar rotas
  • Pode confundir — Next.js espera APIs em app/api/
C

Flat Simples (Evoluir o atual)

Estrutura: Manter a estrutura flat do Next.js como esta, so adicionar as paginas e APIs novas conforme o PRD pede.

app/
  calendario/page.tsx
  dashboard/page.tsx
  pipeline/page.tsx
  referencias/page.tsx
  ideias/page.tsx
  post/[id]/page.tsx
  playbook/page.tsx
  publicados/page.tsx
  api/
    posts/
    references/
    ideas/
    render/
    ...
components/
  calendar/
  post/
  dashboard/
  references/
  ideas/
  pipeline/
lib/
  supabase.ts
  claude.ts
  cloudflare-render.ts
  references.ts
  ideas.ts
  pieces.ts

Pros

  • Menos refatoracao — evolui o que ja existe
  • Padrao classico Next.js App Router
  • Simples de entender

Cons

  • Sem isolamento por conta (sem [accountId] na rota)
  • lib/ vira um saco de gatos com 12 modulos
  • Nao escala bem pra SaaS multi-tenant
  • E basicamente o que voce tem hoje — "remendo"
