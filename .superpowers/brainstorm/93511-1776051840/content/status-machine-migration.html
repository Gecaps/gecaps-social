Maquina de Estados + Plano de Migracao

Como as pecas transitam entre status e como migrar os dados existentes.

Maquina de Estados — Pipeline de Pecas
reference
Referencia cadastrada, aguardando processamento IA
↓ IA processa + gera 5 ideias
idea
Ideia gerada, aguardando avaliacao do operador
↓ operador aprova
idea_approved
Ideia aprovada, IA gera peca automaticamente
↓ IA gera peca completa
in_production
Peca gerada, operador revisa e ajusta
↓ operador aprova final
final_approved
Pronta. Aguardando agendamento ou publicacao.
↓ operador agenda data
scheduled
Agendada no calendario com data e horario
↓ operador publica (download manual)
published
Publicada. Aguardando registro de metricas.

Estados laterais (saem do fluxo principal)

rejected
Reprovada. Vai pra in_adjustment, NAO volta atras.
De: idea, in_production
in_adjustment
Em ajuste. Operador corrige e reenvia pra aprovacao.
Vai pra: in_production
paused
Pausada temporariamente. Pode retomar de qualquer status.
Volta pro status anterior

Transicoes validas (status-machine.ts)

reference     → idea                 (IA processa)
idea          → idea_approved        (operador aprova)
idea          → rejected             (operador rejeita)
idea_approved → in_production        (IA gera peca)
in_production → final_approved       (operador aprova)
in_production → rejected             (operador rejeita)
final_approved → scheduled           (operador agenda)
scheduled     → published            (operador publica)
rejected      → in_adjustment        (automatico)
in_adjustment → in_production        (operador reenvia)
qualquer      → paused               (operador pausa)
paused        → {status anterior}    (operador retoma)
Plano de Migracao — Dados Existentes

Tabelas que mudam

social_accounts → accounts
Renomear. Adicionar campo avatar_url. Dados preservados.

social_posts → pieces
Renomear. Adicionar campos novos (idea_id, creative_brief, visual_direction, slide_structure, published_date, rejection_reason). Migrar status: pending→in_production, approved→final_approved, rejected→rejected, published→published. Remover campos Trello (trello_card_id, trello_list_name).

social_post_versions → piece_versions
Renomear. Adicionar campos novos (creative_brief, visual_direction, slide_structure, change_type).

social_editorial_lines → REMOVER
Substituida pelo Brand Playbook. Dados nao sao migrados.

Tabelas novas

brand_playbooks
1 por conta. Criada vazia, operador preenche.

references
Biblioteca de referencias. Comeca vazia.

ideas
Ideias geradas/manuais. Comeca vazia.

metrics
1 por peca publicada. Comeca vazia.

insights
Insights por conta. Comeca vazia.
⚠ Estrategia de migracao: Uma unica migration SQL no Supabase que renomeia tabelas, adiciona colunas novas, migra status e cria tabelas novas. Os ~20 posts existentes sao preservados como pecas com status migrado. Nenhum dado e perdido.
